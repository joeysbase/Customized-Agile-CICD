# What worked this week?

> In this section list part of the team's process that you believe worked well. "Worked Well" means helped the team be more efficient and/or effective. Try to explain **why** these actions worked well.
### 1. Granular Task Decomposition & Assignment
We addressed last week’s ambiguity by breaking down broader objectives into finer-grained **Issues**. Each task now includes a detailed **Description** and a clear **Assignee**.
* **Why it worked:** This eliminated confusion and reduced communication overhead. By providing a clear roadmap for each contributor, we enabled more effective parallel development and improved overall team velocity.

### 2. Standardized Workflow (PR-to-Issue Mapping)
We streamlined our development pipeline by enforcing a stricter binding between **Pull Requests (PRs)** and their corresponding **Issues**.
* **Why it worked:** This enhanced traceability across the project. It allows team members to quickly understand the "why" behind every code change, simplifying the review process and keeping the project history organized.

### 3. Proactive Architectural Redesign
We took a forward-looking approach by redesigning the **Client-Server architecture** and refactoring core components ahead of the major implementation phase.
* **Why it worked:** Proactively addressing architectural debt has created a more scalable foundation. This preemptive move reduces future friction, ensuring that upcoming features can be integrated more seamlessly and with fewer breaking changes.


# What did not work this week?

> In this section list part of the team's process that you believe did **not** work well. "Not Worked Well" means that the team found these actions to not have a good effect on the team's effectiveness. Try to
 explain **why** these actions did not work well.
### 1. Lack of a Standardized Workflow Pipeline
Although we made progress, we have yet to establish a fully mature and standardized workflow. Team members occasionally followed inconsistent development practices, leading to minor friction in integration.
* **Why it failed:** Without a strictly enforced "Team Playbook" or CI/CD guardrails, individual habits took precedence over collective standards, creating unnecessary back-and-forth during the development cycle.

### 2. Insufficient Code Documentation
The internal documentation for our codebase remains incomplete. In many cases, developers had to explain their code logic verbally rather than others being able to understand it through comments or docs.
* **Why it failed:** This created a "knowledge silo" bottleneck. It hindered true parallel development because team members became dependent on one another for clarification, rather than being able to work autonomously on decoupled modules.

### 3. Absence of Synchronized Weekly Planning
We lacked a fixed, recurring weekly meeting to align on upcoming tasks. Consequently, the breakdown of new features often required multiple ad-hoc rounds of communication to reach clarity.
* **Why it failed:** Without a dedicated "alignment window" at the start of the week, we couldn't reach a consensus on requirements early on. This led to fragmented discussions throughout the week, which disrupted deep work and delayed the start of actual implementation.
